DIYButton ‚Äî Self-contained ESP32 Firmware
Single-flash firmware for an ESP32-C3 that provides:
First-boot captive portal ‚Äî configure WiFi, MQTT, OTA, and button GPIOs via browser
1‚Äì5 configurable buttons ‚Äî GPIO assignments set through the portal, not hardcoded
Home Assistant auto-discovery ‚Äî each button and the LED appear automatically in HA via MQTT
Factory reset ‚Äî hold Button 1 for 5 s to wipe config and re-enter setup mode
---
Hardware
Signal	GPIO	Notes
LED	9 (fixed)	Active high
Button 1	10 (default)	INPUT_PULLUP, active low; also reset hold
Button 2	20 (default)	Optional
Button 3	21 (default)	Optional
Button 4	0  (default)	Optional
Button 5	1  (default)	Optional
GPIO defaults are pre-filled in the setup portal and can be changed freely.
LED is fixed on GPIO9 and cannot be changed via the portal.

Factory Reset
Hold Button 1 (GPIO10 by default, or whatever was configured) for 5 seconds.
LED flashes 3√ó ‚Üí NVS wiped ‚Üí device reboots to AP mode.
The reset hold works in both AP mode and normal mode.

First Boot ‚Äî Setup Portal
On first boot the device starts a WiFi AP named `DIY-Button-Setup-xxxxxxxx` (no password).
Connect from any phone or laptop ‚Äî a captive portal pop-up should appear automatically.
If it doesn't, open `http://192.168.4.1` in a browser.
The portal handles captive-detect probes for iOS/macOS, Android, Windows NCSI, and Firefox,
so the pop-up triggers on most devices without manual navigation.
Portal Fields
Field	Required	Notes
Device Name	Yes	Letters, numbers, `-`, `_` only. Used as the MQTT node ID prefix.
WiFi SSID	Yes	
WiFi Password	No	Leave blank for open networks.
MQTT Broker Host/IP	Yes	
MQTT Port	No	Default 1883.
MQTT Username	No	
MQTT Password	No	
OTA Password	No	Strongly recommended.
Number of Buttons	No	1‚Äì5, default 1.
Button N GPIO	No	GPIO number for each button, pre-filled with defaults.
An API key is auto-generated by the ESP32 on first load (32 random bytes via mbedTLS CSPRNG,
base64-encoded). It is displayed in the portal for reference and stored in NVS; it is not
currently used for authentication but reserved for future use.

Home Assistant Entities
After a successful MQTT connection the device publishes retained auto-discovery payloads.
HA will show:
Entity	Type	Notes
Button 1 ‚Ä¶ Button N	`binary_sensor`	Momentary ON/OFF pulse on press
LED	`light`	ON/OFF, controllable from HA
All entities are grouped under a single HA device identified by `<deviceName>_<MAC>`.
Availability tracking (`online`/`offline`) uses a shared LWT topic.
---
MQTT Topic Layout
All topics are prefixed with `homeassistant/`.
Topic	Description
`binary_sensor/<nodeId>_btnN/config`	HA discovery (retained)
`binary_sensor/<nodeId>_btnN/state`	`ON` pulse then `OFF` on press
`light/<nodeId>/config`	HA discovery (retained)
`light/<nodeId>/state`	Current LED state
`light/<nodeId>/set`	HA ‚Üí device LED command
`status/<nodeId>/availability`	`online` / `offline` (LWT)
`nodeId` = `<deviceName>_<MAC-no-colons-lowercase>`, e.g. `garage-button_a1b2c3d4e5f6`.

Known Gotchas
`"platform":"mqtt"` required ‚Äî HA 2026.x will not process discovery payloads that omit this field. It is included in all payloads.
PubSubClient buffer must be 1024 ‚Äî Discovery payloads are ~570 bytes and silently fail if the buffer is too small. `setBufferSize(1024)` is called in code; `MQTT_MAX_PACKET_SIZE=1024` is also set in `platformio.ini` as a safety net.
Mosquitto password file ‚Äî If MQTT auth fails, confirm the password file was reloaded (`mosquitto_passwd` + broker restart) and that the username/password contain no leading/trailing spaces.
WiFi watchdog ‚Äî If WiFi drops during normal operation the device reboots after 3 s rather than attempting reconnection.
jmdns noise in Arduino IDE ‚Äî Launch from terminal as `arduino 2>/dev/null` to suppress Java mDNS exception spam.
Only one `.ino` file ‚Äî Extra `.ino` files in the sketch folder cause redefinition errors at compile time.
---
Why MQTT Instead of ESPHome Native API?
The ESPHome native API uses a custom noise-protocol-encrypted protobuf framing that would
require implementing ~500+ lines of low-level protocol code in Arduino with high risk of
subtle incompatibilities. MQTT auto-discovery achieves identical HA integration (same entity
types, same device grouping, same availability tracking) with a well-supported library and
no protocol reverse-engineering.
